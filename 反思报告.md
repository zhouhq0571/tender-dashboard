# 招标看板定时任务问题根因分析与解决方案

## 一、问题确认

### 今天定时任务（2026-06-30 7:31）执行结果：
- 生成了 56 个项目
- 但与我昨天手动修复后的 56 个项目**内容完全不同**（1587 行差异）
- 28% 的预算字段为空（16/56）
- 62% 是千里马项目（35/56）
- 版本号显示 v49，日期显示 6月29日（而非6月30日）

### 用户的两个核心问题：

**问题1：定时任务为什么另开窗口？**

**问题2：定时任务为什么覆盖/删除大量项目，而不是增量更新？**

## 二、根因分析

### 根因一：增量更新机制完全失效（最严重）

**问题本质：** 定时任务每次执行都是**从头生成**看板，而不是在现有数据基础上**增量更新**。

**具体表现：**
1. 定时任务没有正确读取现有 index.html 中的项目数据作为基础
2. 每次搜索到的结果直接写入，覆盖现有数据
3. 搜索策略依赖千里马（62%），但千里马返回的结果与用户手动维护的数据不一致
4. 导致大量项目被替换而非保留

**为什么会这样？**
- SKILL.md 的"步骤5：数据合并"虽然写了"保留现有项目+新增项目"，但**定时任务prompt中没有明确执行这个步骤**
- 定时任务prompt中缺少"先读取现有数据"的强制指令
- agent 执行时倾向于"重新生成"而不是"增量更新"
- 70%阈值保护机制只保护项目数量，不保护项目内容质量

### 根因二：数据质量严重下降

**问题表现：**
- 28% 的预算字段为空（因为千里马详情sheet没有预算信息，agent没有从URL获取补充）
- 一些项目只有基本信息（标题、URL），缺少概述、联系人、预算等核心字段
- 这是因为agent在搜索时只获取了表层信息，没有深入获取每个项目的详情

**为什么会这样？**
- 搜索阶段获取大量项目后，agent没有足够时间/资源去逐个获取详情
- 尤其是千里马项目，如果没有读取详情sheet，只能获取基本信息
- 没有设置"信息完整性检查"机制——预算/联系人/概述为空的项目不应该直接纳入

### 根因三：版本号和日期管理混乱

**问题表现：**
- 今天定时任务生成的版本是 v49，日期是 6月29日（应该是 v52，6月30日）
- 说明 deploy.sh 没有正确更新版本号

**为什么会这样？**
- 定时任务中的 agent 可能没有正确执行 deploy.sh 的步骤
- 或者 deploy.sh 中自动更新版本号的逻辑在定时任务环境中没有正确执行
- 更重要的是，定时任务生成的看板没有读取正确的版本号

### 根因四：Prompt 设计缺陷

**问题表现：**
- 定时任务 prompt 中虽然有排除规则，但**没有明确增量更新的步骤**
- "[其余规则保持不变...]"的截断导致 agent 可能跳过关键步骤
- 没有强制要求"先读取现有数据，再增量更新"

**核心设计错误：**
定时任务的设计理念是"每天生成一个新的看板"，而不是"在现有看板上增量更新"。
这是根本性的方向错误。

### 根因五：另开窗口的问题

**问题本质：** 这是 Daimon/Kimi Work 系统的架构限制。
- cron 的 `local_conversation` 模式每次触发都会创建一个新的本地对话
- 每次执行都是零上下文的新会话
- 无法在当前对话窗口中执行

**这不是我能直接修改的。** 但可以考虑替代方案。

---

## 三、正确的每日更新机制设计

### 核心原则：增量更新，绝不覆盖现有数据

```
正确的流程：
1. 读取现有 index.html → 获取当前所有项目（基础数据）
2. 遍历现有项目 → 删除 deadline < 昨天（已过期）的项目
3. 遍历现有项目 → 更新状态变更（如deadline变更、招标方式变更）
4. 搜索新发现项目 → 筛选、补充信息
5. 合并：现有保留项目 + 新发现项目
6. 去重检查
7. 质量门检查
8. 部署
```

### 关键改进点：

1. **强制读取现有数据**：每次执行第一步必须是读取现有 index.html，作为基础数据
2. **增量而非覆盖**：只添加新项目，只删除过期项目，绝不替换现有项目
3. **信息完整性检查**：新项目必须包含预算、截止日期、概述、联系人等核心字段，否则不纳入
4. **版本号正确管理**：每次更新版本号+1，日期正确更新
5. **搜索策略调整**：不要过度依赖单一来源（如千里马62%），要保证多来源平衡
6. **另开窗口问题**：虽然系统限制无法在当前窗口执行，但可以在执行后将结果汇总到当前对话

## 四、用户要求的每日迭代更新机制

用户明确要求：
> "我们每天的定时任务不是从里开始生成看板，而其实是不断增量更新和修订看板的迭代机制"

**正确机制：**
1. **保留**所有现有项目（除了已过期）
2. **更新**状态变更的项目（如deadline变更、二次招标）
3. **补充**现有项目缺失的信息（如预算、联系人）
4. **新增**新发现的高质量项目
5. **绝不**替换现有项目的内容

---

## 五、具体修复方案

### 方案1：修改定时任务prompt，强制增量更新

在定时任务prompt中明确：
```
## 强制步骤：增量更新（★★★ 核心，不可跳过）

### 步骤1：读取现有数据
1. 读取本地 /Users/zhouhq/Documents/kimi/workspace/bidding-daily/index.html
2. 提取所有现有项目，记录项目总数、版本号、日期
3. 这是你的基础数据，**绝不能覆盖或替换**

### 步骤2：删除已过期项目
- 遍历所有现有项目，检查截止日期
- deadline < 昨天（即早于今天的前一天）→ 从数据中删除
- 记录删除了多少个项目

### 步骤3：搜索新发现项目
- 执行四阶段搜索（阶段0→A→B→C）
- 对新发现项目进行筛选和排除
- 对新发现项目补充完整信息（预算、联系人、概述等）
- **信息不完整的项目（预算、截止日期、概述缺失）不纳入**

### 步骤4：合并
- 保留的现有项目 + 新发现项目 = 新的项目列表
- 去重检查（同一公司+项目名称）
- 重新编号

### 步骤5：质量门检查
- 检查是否有信息不完整的项目
- 检查是否有已过期项目未删除
- 检查是否有重复项目
- 不通过则停止部署

### 步骤6：部署
- 更新版本号（vX+1）
- 更新日期（今天）
- 更新时间（执行时间）
- 执行 deploy.sh
```

### 方案2：信息完整性检查（新项目准入门槛）

```
新项目必须包含以下字段（不能为"-"或空）：
1. 项目概述（overview）
2. 预算金额（budget）
3. 投标截止日期（deadline）
4. 招标方式（method）

缺少任一字段的新项目 → 不纳入，标记为"待补充"写入 LAST_RUN_STATUS
```

### 方案3：另开窗口问题的替代方案

虽然无法在当前窗口执行定时任务，但可以：
1. 定时任务执行后，将结果写入 LAST_RUN_STATUS.md
2. 在当前对话中，我可以通过查看 LAST_RUN_STATUS.md 来确认执行结果
3. 如果发现问题，在当前对话中手动修复

或者考虑：不使用定时任务，改为每天在当前对话中手动执行一次更新（但这违背了自动化原则）。

**最实际的方案：** 保留定时任务（系统限制），但改进prompt确保质量，同时在当前对话中我主动检查定时任务执行结果。

## 七、总结

**用户的愤怒完全合理。**

定时任务的设计从根本上就是错的：
- 不是增量更新，而是从头生成
- 没有保护现有数据
- 没有信息完整性检查
- 导致大量项目被替换、数据丢失、质量下降

**必须立即修复。**

修复后的机制：
1. 读取现有数据 → 基础
2. 删除过期 → 精简
3. 搜索新发现 → 补充
4. 合并 → 增量
5. 质量门 → 保证
6. 部署 → 输出

这样用户每天看到的是：昨天的项目 + 今天的新发现，而不是一个全新的、信息不全的看板。
