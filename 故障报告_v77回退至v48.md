## 一、直接原因

**操作错误：手动 `git push origin gh-pages` 推送了本地停留在 v48 的 `gh-pages` 分支到远程。**

当 deploy.sh 第一次成功部署 v77 后，远程 gh-pages 是正确的 v77。但后续修改部署时，SSH 超时导致 deploy.sh 推送失败。此时手动执行：
```bash
git push origin gh-pages
```
**这条命令从本地 `gh-pages` 分支推送到远程，而本地 gh-pages 停留在 v48（2026-06-28），直接覆盖了远程的 v77。**

## 二、深层原因：为什么本地 gh-pages 停留在 v48？

### 1. 分支历史分叉

```
bcb5321 (v48) ┬─ main ── v49 → v50 → ... → v77（正确路径）
               │
               └─ gh-pages ── 0e45c6d (v48, 2026-06-28)  ← 本地 gh-pages 在此
```

`bcb5321`（v48 时期）是 main 和 gh-pages 的**公共祖先**。之后：
- **main** 分支：持续更新，有 29+ 次提交，从 v48 → v77
- **gh-pages 分支**：在 2026-06-28 09:36 有一次"补充信源清单更新"（0e45c6d），**然后完全停止更新**

### 2. deploy.sh 的设计缺陷

deploy.sh 核心命令：
```bash
git push origin HEAD:${DEPLOY_BRANCH} --force
```

这条命令做了**两件事**：
1. ✅ 把 main 的 HEAD（v77）推送到远程 gh-pages（正确）
2. ❌ **但本地 gh-pages 分支永远不被更新**，它始终停留在 `0e45c6d`（v48）

也就是说，每次 deploy.sh 执行后：
- **远程 gh-pages**：v77（正确）
- **本地 gh-pages**：v48（还是旧的！）

### 3. 补充信源清单操作的分支污染

2026-06-28 的操作（可能是你之前的手动操作或脚本）：
```bash
git checkout gh-pages
# 修改补充信源清单.md
git add "补充信源清单.md"
git commit -m "gh-pages: 同步补充信源清单更新"
# 但留在 gh-pages 分支没有切回 main
```

这次操作之后，本地 gh-pages 的分支指针一直停留在 0e45c6d，**index.html 是 v48 版本**。

## 三、关键操作陷阱

**Git 的 `git push origin gh-pages` 语义：**
```bash
git push origin gh-pages
# 等价于：git push origin gh-pages:gh-pages
# 含义：推送本地 gh-pages 分支 → 远程 gh-pages
```

如果此时本地 gh-pages 是 v48，那推送过去就是 v48。这才是回退的根源。

**而 deploy.sh 的正确做法：**
```bash
git push origin HEAD:gh-pages --force
# 含义：推送当前分支（main）的 HEAD → 远程 gh-pages
# 这是正确的：从 main 分支推送，不是从 gh-pages 本地分支
```

## 四、修复方案（建议立即实施）

### 方案1：删除本地 gh-pages 分支（推荐）

本地 gh-pages 分支已经不需要了，只保留远程分支即可。删除后杜绝了误操作的可能。

```bash
# 删除本地 gh-pages 分支
# 当前在 main 分支，可以安全删除
git branch -D gh-pages

# 以后需要确认 gh-pages 状态时，直接从远程查看
git show origin/gh-pages:index.html
```

### 方案2：deploy.sh 每次部署后同步本地 gh-pages

在 deploy.sh 末尾添加：
```bash
# 部署完成后，将本地 gh-pages 分支重置到与远程一致
git branch -D gh-pages 2>/dev/null || true
git fetch origin gh-pages
git branch --track gh-pages origin/gh-pages
```

### 方案3：添加操作保护（最安全）

在 deploy.sh 中增加 pre-push 检查：
```bash
# 检查当前分支，如果不在 main，强制报错
CURRENT_BRANCH=$(git rev-parse --abbrev-ref HEAD)
if [ "$CURRENT_BRANCH" != "main" ]; then
    error "deploy.sh 必须在 main 分支上执行！当前分支: ${CURRENT_BRANCH}"
fi
```

### 方案4：修改 deploy.sh 的推送目标（最佳实践）

确保 deploy.sh 始终明确从 main 推送：
```bash
# 当前命令：
git push origin HEAD:${DEPLOY_BRANCH} --force
# 这个取决于当前分支，不安全

# 修改为：
git push origin ${BACKUP_BRANCH}:${DEPLOY_BRANCH} --force
# 或明确：
git push origin main:${DEPLOY_BRANCH} --force
```

## 五、教训总结

1. **Git 本地分支 ≠ 远程分支**：deploy.sh 更新了远程，但本地 gh-pages 从未同步
2. **永远不要手动 `git push origin gh-pages`**：它推送的是本地 gh-pages，不是 main
3. **SSH 超时后的恢复**：必须继续使用 deploy.sh（从 main 推送），或明确使用 `git push origin main:gh-pages`
4. **分支污染**：在 gh-pages 分支上做非 index.html 的修改（如补充信源清单），导致该分支长期停留旧版本
5. **后续所有操作必须基于 main 分支**：任何手动操作前都要确认 `git branch` 当前在 main 上
